# AI REPORT DRAFT

## 프로젝트 한 줄 정의

MajorMap AI는 대학생의 수강 이력, 학번별 커리큘럼, 전공 조합을 바탕으로 적합한 진로를 추천하고, 부족한 역량과 보완 과목을 안내하는 규칙 기반 진로 설계 서비스다.

---

## 문제 정의

복수전공·심화전공 대학생은 자신이 들은 과목이 어떤 진로와 연결되는지 파악하기 어렵다. 특히 학번별 커리큘럼 차이와 전공 조합의 다양성 때문에, 일반적인 진로 테스트나 선배 경험만으로는 현재 상태를 설명하기 어렵다.

이 프로젝트는 "내가 이미 들은 과목이 어떤 역량으로 해석되는가"와 "그 결과 어떤 진로가 적합한가"를 연결하는 문제를 다룬다.

---

## 왜 교육 현장 pain point인지

교육 현장에서 학생은 단순히 관심 직무를 아는 것보다, 현재 이수 상태를 기준으로 어떤 진로가 가능한지 알고 싶어 한다. 하지만 실제 상담에서는 다음 문제가 자주 발생한다.

- 이수 과목이 직무 역량으로 어떻게 해석되는지 알기 어렵다.
- 학번별 개설 과목 차이 때문에 동일 학과 학생끼리도 경로가 달라진다.
- 복수전공 조합이 어떤 진로에서 강점이 되는지 직관적으로 설명받기 어렵다.
- 진로 추천이 나오더라도 근거가 불명확하면 신뢰하기 어렵다.

MajorMap AI는 과목 태그와 전공 배경을 기반으로 추천 근거를 구조화해 이 문제를 완화하려고 했다.

---

## MVP 범위와 의도적 제한

이번 제출 버전은 다음 범위에 한정한다.

- 대학: 1개 대학 가정
- 학과군: 컴퓨터공학, 경영학
- 학번: 2023학번, 2024학번
- 진로: 백엔드 개발자, 프로덕트 매니저, 데이터 분석가, 솔루션 컨설턴트
- 과목 데이터: 15개 샘플

의도적으로 제외한 항목은 다음과 같다.

- 실제 학사 시스템 연동
- 로그인/회원가입
- 전 학과 확장
- 시간표 충돌 계산
- 실제 채용 공고 연동

이 제한은 기능을 줄인 것이 아니라, 작은 범위에서 추천 근거의 해석 가능성과 제출 안정성을 확보하기 위한 선택이다.

---

## 시스템 구조

서비스는 Next.js App Router 기반으로 구성되어 있다.

- `app/api/recommend/route.ts`
  - 학생 입력을 받아 규칙 기반 추천 결과를 반환한다.
- `app/api/explain/route.ts`
  - 추천 결과에 대한 설명을 생성하며, 실패 시 fallback으로 전환한다.
- `lib/sample-data.ts`
  - 과목, 진로, 데모 프로필, 태그 레이블을 관리한다.
- `lib/scoring.ts`
  - 점수 계산과 추천 이유 생성 규칙을 담당한다.
- `lib/recommendation.ts`
  - 상위 추천 결과를 조합한다.
- `lib/llm.ts`
  - OpenAI 호출과 fallback 설명 생성 로직을 담당한다.
- `components/*`
  - 입력 폼, 결과 카드, explain 패널을 렌더링한다.

핵심 구조는 추천 로직이 LLM 없이도 동작하고, AI는 설명 계층에만 제한적으로 연결된다는 점이다.

## 추천 로직 구조

추천 로직은 과목 태그 기반의 규칙형 점수 계산으로 이루어져 있다.

1. 학생이 수강한 과목에서 태그를 수집한다.
2. 각 진로의 필수 태그와 선택 태그를 비교한다.
3. 관심 키워드와 진로 alias를 비교해 키워드 보너스를 계산한다.
4. 주전공/복수전공이 선호 전공과 맞으면 전공 보너스를 반영한다.
5. raw score를 100점 기준으로 정규화한다.

현재 구현된 가중치는 다음과 같다.

- 필수 태그: 태그당 30점
- 선택 태그: 태그당 10점
- 키워드 매칭: 매칭당 6점
- 주전공 보너스: 10점
- 복수전공 보너스: 6점

추천 결과에는 점수만이 아니라 다음 정보가 함께 제공된다.

- score breakdown
- 보유 역량
- 부족 역량
- 추천 이유
- 보완 과목

즉, 이 서비스는 결과 이름만 보여주는 것이 아니라, 왜 그런 결과가 나왔는지 해석할 수 있도록 설계되어 있다.

## explain fallback 구조

explain 기능은 보조 계층이다.

- `OPENAI_API_KEY`가 없으면 fallback 설명을 반환한다.
- 외부 API 호출이 실패하면 fallback 설명을 반환한다.
- 응답 형식이 기대값과 다르면 fallback 설명을 반환한다.
- 서버 전용 코드로 유지해 클라이언트에 키가 노출되지 않도록 한다.

fallback 설명은 추천 결과 자체에 포함된 정보를 사용해 구성한다.

- 적합도 headline
- 추천 요약
- 추천 근거
- 보완 포인트
- 간단한 로드맵

이 구조의 목적은 AI 설명이 실패해도 추천 핵심 흐름이 깨지지 않게 만드는 것이다.

## AI와 사람의 역할 분담

AI는 구현과 정리의 초안을 빠르게 만들고, 사람은 의미와 타당성을 검토하는 방식으로 협업했다.

AI가 주로 기여한 영역:

- 코드베이스 분석과 작업 계획 정리
- 타입 구조 설계
- 점수 계산 함수 정리
- 추천 로직 리팩터링
- explain fallback 구조 구현
- 문서 초안 작성

사람이 직접 결정한 영역:

- 과목-진로 태그 매핑 타당성
- 점수 가중치 균형
- 데모 프로필 A/B/C의 결과 차이
- fallback 문구의 자연스러움
- public repo 기준 보안 규칙

즉, AI는 생산성을 높였고, 최종 의미 판단은 사람이 담당했다.

## 실제 AI 협업 방식

실제 개발에서는 Cursor Agent를 중심으로 협업했다.

1. 현재 저장소 상태를 먼저 읽는다.
2. 필요한 작업을 작은 단위로 나눈다.
3. AI가 코드와 문서 초안을 제안한다.
4. 사람이 범위와 타당성을 검토한다.
5. `npm run lint`, `npm run build`, 실제 흐름 확인 결과를 기준으로 반영 여부를 결정한다.

협업 기록은 `docs/AI_COLLAB.md`에 남겨 두었다.

## public repo / env / 보안 대응

이번 제출은 public GitHub 저장소를 전제로 정리했다.

보안 대응 원칙은 다음과 같다.

- 실제 API 키는 코드에 하드코딩하지 않는다.
- `.env*` 파일은 Git 추적 대상에서 제외한다.
- `.env.example`에는 placeholder만 남긴다.
- `OPENAI_API_KEY`는 선택 환경변수로만 문서화한다.
- API 키가 없어도 추천 핵심 기능은 유지된다.

이 프로젝트는 AI 설명 기능이 없어도 동작해야 한다는 구조를 통해 외부 비밀값 의존도를 낮췄다.

## 한계와 향후 개선

현재 한계는 다음과 같다.

- 대학 1곳, 학과군 2개, 학번 2개에 한정된 샘플 데이터 구조다.
- 과목 수와 진로 수가 제한적이다.
- 실제 상담 품질 검증 데이터가 없다.
- 설명 기능은 보조 계층이며, 모델 품질을 핵심 가치로 두지 않는다.

향후 개선 방향은 다음과 같다.

- 실제 대학 커리큘럼 데이터 확장
- 학과군과 학번 범위 확대
- 보완 과목 추천 규칙 정교화
- 사용자 검증을 통한 점수 규칙 재조정
